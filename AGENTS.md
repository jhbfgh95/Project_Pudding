# ServerTest 작업 가이드

## 프로젝트 개요

- 목적: Unreal Engine 멀티플레이어와 Steam 연동을 학습하기 위한 Listen Server 기반 프로토타입입니다.
- 구현 비중: 그래픽, 머티리얼 등 에셋 작업을 제외한 기능의 약 90%를 C++로 구현합니다.
- 네트워크 구조: Listen Server
- 온라인 서비스: Steam Online Subsystem 연동

## 현재 목표

1. Steam 연동과 세션(방) 생성·참여를 구현합니다.
2. 아이템 획득 상태가 모든 접속자에게 일관되게 동기화되도록 구현합니다.
3. 점프 동작이 서버 권한을 기준으로 자연스럽게 동기화되도록 구현합니다.
4. 피해 처리와 체력 등 최소한의 간이 전투를 구현합니다.

## 기능 기획 문서

- 아이템, 상호작용, 인벤토리, 점프 관련 조사·설계·구현을 시작하기 전에 반드시 [ITEM_INTERACTION_DESIGN.md](ITEM_INTERACTION_DESIGN.md)를 확인합니다.
- 해당 문서는 사용자가 합의한 프로토타입 구조와 범위를 기록한 기준 문서입니다. 구현 방향을 바꿔야 할 때는 먼저 사용자에게 영향과 대안을 설명하고 동의를 받습니다.

## 작업 원칙

- **사용자의 구현 동의가 먼저입니다.** 조사, 설계 제안, 코드 설명, 원인 분석은 바로 수행할 수 있지만, 실제 코드·설정·에셋 파일 변경은 구현 범위와 방식을 설명하고 사용자의 명시적인 동의를 받은 뒤 진행합니다.
- 한 번에 큰 기능을 완성하려 하지 않고, 학습 가능한 작은 단위로 나누어 제안하고 구현합니다.
- 구현 전에는 다음을 간단히 설명합니다.
  - 무엇을 구현하는지
  - 왜 해당 구조와 Unreal 네트워크 기능을 쓰는지
  - 수정될 파일과 예상되는 결과
- 구현 후에는 변경 내용, 핵심 C++ 개념, 네트워크 흐름, 직접 확인할 테스트 방법을 NOTES.md에 기록합니다.
- 사용자가 이해하기 쉽게 용어를 풀어 설명하고, 필요한 경우 짧은 코드 주석을 한국어로 남깁니다.

## Unreal 네트워크 규칙

- 게임 결과를 바꾸는 판정(아이템 획득, 피해, 체력 변경 등)은 서버가 최종 권한을 가집니다.
- 클라이언트는 요청을 보내고, 서버는 유효성을 확인한 뒤 상태를 변경합니다.
- 공유해야 하는 상태는 Replication을 사용하며, 상태 변경 후 클라이언트 반응이 필요하면 OnRep 함수를 사용합니다.
- RPC에는 호출 방향(Server/Client/NetMulticast), Reliable 사용 이유, 호출 주체를 명확히 합니다.
- Listen Server와 원격 클라이언트 양쪽에서 테스트합니다. 호스트에서만 정상인 경우 완료로 판단하지 않습니다.

## Steam 및 세션 작업 규칙

- Steam 관련 설정과 `DefaultEngine.ini` 변경은 영향 범위를 먼저 설명하고 동의를 받은 뒤 적용합니다.
- 세션 생성, 검색, 참가, 종료의 각 단계를 로그로 확인할 수 있게 구성합니다.
- 로컬 테스트와 Steam 실행 환경의 차이를 NOTES.md에 기록합니다.

## 완료 기준

- 기능은 Listen Server 호스트와 최소 한 명의 원격 클라이언트에서 재현·확인합니다.
- 실패 시 로그 위치, 재현 절차, 관찰 결과를 NOTES.md에 남깁니다.
- 미확인 항목은 완료로 표시하지 않고, 다음 확인 절차를 제안합니다.
